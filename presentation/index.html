01. Overview
Introduction & Problem Context
SPRAMS
The Challenge
  • Fragmented Repositories: Student reports, thesis archives, and faculty research are scattered across Google Drives, email logs, and physical drives.
  • Loss of Institutional Knowledge: Historical projects become inaccessible to junior batches, leading to redundant work.
  • Manual Inefficiencies: Supervisor feedback loop is paper-based or disjointed; grading rubrics are manually structured.
The Solution (SPRAMS)
  • Centralized Archive Hub: Single point of truth for project reports, software code references, and research publications.
  • AI-Enhanced Metadata: Automated processing utilizing Claude AI to parse documents, draft keyword tags, and suggest abstract improvements.
  • Dedicated Workspaces: Custom workspaces for Students to upload drafts, and Lecturers to critique/evaluate work online.

02. Requirements
Requirement Analysis: Non-Functional
SPRAMS
Security & Cryptography
  • JWT Auth: Dual-token flow with HttpOnly cookies preventing XSS and CSRF token theft.
  • Helmet Headers: Inject secure HTTP response headers to harden the browser client against common exploits.
  • Rate Limiter: API-level limits protecting login and upload gateways from brute-force scripts.
Performance Optimization
  • AI Caching: Claude's generated summary objects are cached in the database to prevent duplicate API requests and fees.
  • Database Indexes: Combined indexes on `department`, `academicYear`, and Text Indexing on search query target fields.
  • Asset Size Limit: Optimized file pipeline limiting single uploads to 100MB to sustain bandwidth.
Reliability & Audit Trails
  • Centralized Logging: Winston logger creates daily rotating log files separating `error.log` and `combined.log`.
  • Database Validation: Strict Mongoose schemas with validation guards avoiding database corruption.
  • API Error Handler: Uniform global middleware transforming server failures into clean JSON responses.

03. System Design
System Design: Database Schema
SPRAMS
User Collection Users
_id ObjectId (PK)
name String
email String (Unique)
role Enum ('student','lecturer','admin')
approvalStatus Enum ('pending','approved')
Project Collection Projects
_id ObjectId (PK)
title String (Text indexed)
department String
academicYear String
members Array ([name, regNo])
createdBy ObjectId (FK -> User)
Workspace Submission WorkspaceSubmissions
submissionID String (PK)
projectID ObjectId (FK -> Project)
submissionType Enum ('Proposal','Progress','Final')
submittedBy ObjectId (FK -> Student)
Evaluation Collection Evaluations
submissionId ObjectId (Polymorphic PK)
evaluatedBy ObjectId (FK -> User)
marks / grade Number / String (Letter grade)
approvalStatus Enum ('Approved','Revision')
